# 需求概述

## 背景

当前团队在多个平台上协同推进工作：

- `Lark 多维表`
  - `A1`：支持工单表
  - `A2`：需求表
- `Meegle`
  - `B1`：需求任务
  - `B2`：产线 Bug
- `GitHub`
  - `P1`：开发 Pull Request

现有流程中，业务同事从 `A1` 提交支持工单。部分工单会被直接处理，部分会转入 `B2` 作为产线问题，部分会沉淀到 `A2` 作为需求；之后需求会被定期评审，进入 `B1` 进行研发执行。研发完成后再进入测试、PR、Code Review 和上线流程。

## 当前痛点

1. `A1 -> B2`、`A2 -> B1` 属于跨平台创建，当前主要依赖人工复制、整理、建单。
2. 需求进入开发前，文档经常不完整，导致研发需要反复追问。
3. 开发交付给测试时，说明材料经常不完整，影响测试理解和推进。
4. PM 做项目统筹时需要在多个平台之间手工汇总信息，缺少高效的即时分析工具。

## 产品目标

构建一个面向 PM / 需求负责人的跨平台协同助手，解决以下三个核心问题：

1. `跨平台半自动建单`
2. `A1/A2 智能分析补全`
3. `PM 即时分析`

## 核心设计原则

1. 不替代现有平台，只做协同增强。
2. 不维护业务镜像，尽量实时拉取最新数据。
3. AI 能力集中在服务端，以 `agents + skills` 形式组织。
4. 浏览器插件保持轻量，只做页面侧入口、上下文采集与结果展示。
5. 一切执行动作都保留人工确认。

## 目标用户

第一期主用户仅为：

- `项目经理`
- `需求负责人`

开发、测试和业务同事在第一期中不是主操作用户，但会间接受益于更高质量的任务描述和更顺畅的协同流程。

## 一期产出定义

第一期产品定义为：

> 一个以浏览器插件为主入口、以服务端 Agent / Skill 编排为核心、面向 PM 的跨平台协同助手。

## 非目标

以下内容不在第一期范围内：

- 独立 PM 看板产品
- Lark / Meegle / GitHub 业务镜像
- PR 描述生成
- 全自动流程推进
- 多团队、多租户

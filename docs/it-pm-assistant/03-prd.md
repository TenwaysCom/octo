# PRD

## 1. 产品定位

一个面向 PM / 需求负责人的跨平台协同助手，帮助用户在 Lark、Meegle、GitHub 之间更高效地推进工单、需求和项目分析。

核心价值为：

> 跨平台半自动建单 + A1/A2 智能分析补全 + PM 即时分析

## 2. 目标用户

- 项目经理
- 需求负责人

## 3. 用户核心场景

### 场景一：支持工单分流

用户在 `Lark A1` 中查看工单，希望快速判断：

- 是否直接处理
- 是否应该转为 `Meegle B2`
- 是否应该沉淀为 `Lark A2`
- 当前信息是否足以进入下一步

### 场景二：需求可执行化

用户在 `Lark A2` 中查看需求，希望快速产出一份适合研发进入 `Meegle B1` 的任务说明。

### 场景三：即时项目分析

用户希望快速了解一个项目、版本或时间窗口内的推进状态、阻塞点和描述缺口，而不需要在多个平台间来回切换和手工汇总。

## 4. 功能清单

### 4.1 Lark A1 页面能力

- 智能分析
- 缺失信息提示
- 转 Meegle Bug
- 转需求池 A2
- 补全描述

### 4.2 Lark A2 页面能力

- 智能补全需求
- 评审建议
- 生成研发说明
- 转 Meegle 需求

### 4.3 Meegle 页面能力

- 查看来源上下文
- 补全任务说明
- 生成提测说明

### 4.4 GitHub 页面能力

- 读取当前用户 `GitHub ID`
- 读取 PR / review / merge 状态
- 为 PM 即时分析提供交付状态数据

### 4.5 全局能力

- PM 即时分析
- 当前用户识别
- 用户映射校验
- 半自动创建确认

## 5. 非功能要求

- 插件需要兼容主流 Chromium 浏览器，后续可扩展更多兼容浏览器。
- 所有写操作必须人工确认。
- 对平台接口异常具备清晰降级提示。
- AI 输出必须结构化并经过 schema 校验。

## 6. 非目标

- 不做项目看板页面
- 不做业务数据镜像
- 不做 PR 描述生成
- 不做全自动流程推进
- 不做多租户

## 7. 成功标准

- PM 从 A1/A2 创建 B2/B1 的平均时间下降
- B1/B2 描述完整度提升，减少来回补充
- PM 可以通过一次分析得到跨平台结论
- 产品形态保持轻量，不形成新的重型项目管理系统

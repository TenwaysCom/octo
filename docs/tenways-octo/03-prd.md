# PRD

## 1. 产品定位

一个面向 PM / 需求负责人的跨平台协同助手，帮助用户在 Lark、Meegle、GitHub 之间更高效地推进工单、需求和项目分析。

核心价值为：

> 跨平台半自动建单 + A1/A2 智能分析补全 + PM 即时分析

## 2. 目标用户

- 项目经理
- 需求负责人

## 3. 用户核心场景

### 场景一：支持工单分流

用户在 `Lark A1` 中查看工单，希望快速判断：

- 是否直接处理
- 是否应该转为 `Meegle B2`
- 是否应该沉淀为 `Lark A2`
- 当前信息是否足以进入下一步

### 场景二：需求可执行化

用户在 `Lark A2` 中查看需求，希望快速产出一份适合研发进入 `Meegle B1` 的任务说明。

### 场景三：即时项目分析

用户希望快速了解一个项目、版本或时间窗口内的推进状态、阻塞点和描述缺口，而不需要在多个平台间来回切换和手工汇总。

## 4. 功能清单

### 4.1 Lark A1 页面能力

- 智能分析
- 缺失信息提示
- 转 Meegle Bug
- 转需求池 A2
- 补全描述

### 4.2 Lark A2 页面能力

- 智能补全需求
- 评审建议
- 生成研发说明
- 转 Meegle 需求

### 4.3 Meegle 页面能力

- 查看来源上下文
- 补全任务说明
- 生成提测说明

### 4.4 GitHub 页面能力

- 读取当前用户 `GitHub ID`
- 读取 PR / review / merge 状态
- 为 PM 即时分析提供交付状态数据

### 4.5 全局能力

- PM 即时分析
- 当前用户识别
- 用户映射校验
- 半自动创建确认

#### 4.5.1 用户身份识别与同步

插件需要能够获取并展示当前用户的跨平台身份信息：

**身份信息获取：**
- **Lark User ID**: 从 Lark 页面获取当前登录用户的 Lark ID
- **Meegle User Key**: 从 Meegle 页面获取当前登录用户的 User Key

**身份信息展示：**
- 在插件 popup 中显示当前用户的 Lark User ID 和 Meegle User Key
- 显示身份绑定状态（已绑定/未绑定）
- 支持手动刷新身份信息

**身份信息同步：**
- 将获取的身份信息同步到后端用户表
- 建立同一用户在不同平台间的身份映射关系
- 支持身份变更时自动更新用户表

**数据流程：**
```
用户打开 Lark/Meegle 页面
    ↓
Content Script 提取用户身份信息
    ↓
Background Script 接收并缓存
    ↓
Popup 显示身份信息
    ↓
后端 API 同步到用户表
```

#### 4.5.2 Chrome 插件 Popup UI 设计

插件 Popup 根据当前页面类型显示不同的 UI 布局：

**页面类型检测：**
- Meegle 页面：`project.larksuite.com` 或 `meegle.com`
- Lark 页面：`*.larksuite.com` 或 `*.feishu.cn`（不包括 Meegle）
- 不支持页面：其他所有页面

**UI 布局规则：**

| 页面类型 | 显示内容 |
|---------|---------|
| Meegle | 授权板块 → Meegle 功能板块（授权后）→ 授权板块（底部） |
| Lark | 授权板块 → Lark 功能板块（授权后）→ 授权板块（底部） |
| 不支持 | 仅显示"当前页面不支持"提示 |

**授权板块设计：**
```
┌─────────────────────────────┐
│ 🔐 授权状态                  │
├─────────────────────────────┤
│ Meegle User  [状态] [授权]  │
│ Lark User    [状态] [授权]  │
└─────────────────────────────┘
```

- **Meegle User**: 显示 Meegle User Key 或授权状态
- **Lark User**: 显示 Lark User ID 或授权状态
- **授权按钮**: 点击后检查授权状态，需要时跳转到登录页面

**功能板块设计：**

Meegle 功能板块：
```
┌─────────────────────────────┐
│ 🔷 Meegle 功能               │
├─────────────────────────────┤
│ [查看来源上下文]             │
└─────────────────────────────┘
```

Lark 功能板块：
```
┌─────────────────────────────┐
│ 📋 Lark 功能                 │
├─────────────────────────────┤
│ [分析当前页面]               │
│ [生成草稿]                   │
│ [确认创建]                   │
└─────────────────────────────┘
```

**授权流程：**
1. 用户打开插件 Popup
2. 插件检测当前页面类型
3. 显示授权板块，展示当前授权状态
4. 用户点击授权按钮
5. 检查授权状态，如果需要登录则跳转到登录页面
6. 授权成功后返回，显示功能板块

**交互规则：**
- 仅在用户点击授权按钮后才跳转到登录页面
- 授权成功后自动刷新状态
- 底部授权板块提供重新授权功能

## 5. 非功能要求

- 插件需要兼容主流 Chromium 浏览器，后续可扩展更多兼容浏览器。
- 所有写操作必须人工确认。
- 对平台接口异常具备清晰降级提示。
- AI 输出必须结构化并经过 schema 校验。

## 6. 非目标

- 不做项目看板页面
- 不做业务数据镜像
- 不做 PR 描述生成
- 不做全自动流程推进
- 不做多租户

## 7. 成功标准

- PM 从 A1/A2 创建 B2/B1 的平均时间下降
- B1/B2 描述完整度提升，减少来回补充
- PM 可以通过一次分析得到跨平台结论
- 产品形态保持轻量，不形成新的重型项目管理系统

# Tenways Octo 插件使用文档

本文档面向日常使用 Tenways Octo 插件的 PM、需求负责人和协作同学，重点说明安装、授权、自动更新，以及目前已经落地的几个核心能力。

## 1. 插件能做什么

Tenways Octo 是一个运行在浏览器里的协同插件，主要工作方式是：

- 在 `Lark` 页面识别当前记录，并把记录创建到 `Meegle`
- 在 `Meegle` 页面反推更新 `Lark`
- 在 `GitHub PR` 页面反查关联的 `Meegle` 工作项
- 在 `Meegle` 页面按工作项信息创建 `GitHub` 分支
- 提供一个带历史会话的 `AI 聊天` 入口，方便继续追问和分析

当前插件支持识别三类页面：

- `Lark / Feishu` 页面
- `Meegle` 页面
- `GitHub Pull Request` 页面

如果当前标签页不在以上范围内，插件会显示为不支持页面。

## 2. 安装

### 2.1 启动服务端

插件依赖后端服务提供鉴权、工作流编排和第三方 API 调用能力。先确认服务端已经启动，并且可以访问你配置的 `SERVER_URL`。

本地开发常用命令：

```bash
pnpm --dir server dev
```

如需本地构建插件：

```bash
pnpm --dir extension install
pnpm --dir extension build
```

构建产物目录：

```text
extension/.output/chrome-mv3/
```

### 2.2 安装到 Chrome

1. 打开 `chrome://extensions/`
2. 打开右上角“开发者模式”
3. 点击“加载已解压的扩展程序”
4. 选择 `extension/.output/chrome-mv3/`
5. 安装完成后，把插件固定到浏览器工具栏，方便后续使用

### 2.3 首次检查设置

打开插件后，先进入右侧 `设置` 页签，重点确认：

- `Server URL` 是否指向正确的 Octo 服务
- `Lark Callback URL` 是否已由服务端正确下发
- `MEEGLE Plugin ID` 是否已由服务端正确下发

其中：

- `Server URL` 可手动修改并保存
- `Lark Callback URL` 和 `MEEGLE Plugin ID` 为只读，通常由服务端 `/api/config/public` 下发

## 3. 授权

插件只有在 `Lark` 和 `Meegle` 都授权完成后，`自动化` 和 `聊天` 两个页签才可正常使用。

### 3.1 查看授权状态

打开插件右侧 `个人` 页签，可以看到：

- `Master User ID`
- `Lark User Name / ID / Email`
- `Meegle User Key`
- `Lark` 与 `Meegle` 当前授权状态

### 3.2 Lark 授权

在 `个人` 页签点击 `Lark` 对应的 `授权` 或 `重新授权`：

1. 插件会先向服务端查询当前 Lark token 状态
2. 如果服务端还能刷新 token，会直接复用
3. 如果需要重新 OAuth，插件会自动打开 Lark 授权页面
4. 授权回跳成功后，插件会自动更新状态

说明：

- 当前实现会请求一组固定 scope，由服务端配置下发
- 授权完成后，插件会自动尝试补齐 Lark 用户资料

### 3.3 Meegle 授权

在 `个人` 页签点击 `Meegle` 对应的 `授权`：

1. 建议当前浏览器标签页就是已经登录的 `Meegle` 页面
2. 插件会在当前页向 Meegle 申请一次性 `auth code`
3. 再由服务端把 `auth code` 换成用户 token
4. 成功后，插件会把 Meegle 状态标记为已就绪

注意：

- 服务端不会接收浏览器原始 cookie
- `auth code` 是一次性凭证
- 如果插件提示 `Meegle 授权失效`，重新走一遍授权即可
- 如果 Lark 和 Meegle 账号存在身份冲突，插件会阻止继续授权

## 4. 自动热更新

插件已经接入自托管更新检查，但安装仍然受 Chrome 限制，需要手动替换扩展包。

### 4.1 更新检查时机

插件会在以下时机静默检查新版本：

- 浏览器启动后
- 插件 popup 初始化时
- 后台定时任务，默认每 24 小时一次

### 4.2 用户可见表现

如果服务端返回了更高版本：

- 插件图标会出现红色角标
- 打开插件后，顶部会出现更新横幅
- 横幅里会显示 `当前版本 -> 新版本`
- 如果服务端配置了 `releaseNotes`，也会直接展示

当前横幅提供两个动作：

- `立即更新`
- `忽略此版本`

### 4.3 更新方式

点击 `立即更新` 后，插件会通过浏览器下载扩展压缩包。下载完成后，需要你自己去 `chrome://extensions/` 替换为新版本包。

这不是 Chrome Web Store 的自动安装更新，因此更准确地说是：

- 自动检查更新
- 自动下载更新包
- 手动完成安装替换

如果点击 `忽略此版本`，该版本会被临时忽略，约 24 小时后再次出现。

## 5. Lark 创建 Meegle

当前有两种使用方式。

### 5.1 在 Lark 记录详情页直接创建

当你打开受支持的 `Lark Base / Wiki 记录详情页` 时，页面内会注入一个按钮。按钮文案会根据 `Issue 类型` 自动变化，例如：

- `创建 Meegle User Story`
- `创建 Meegle Tech Task`
- `创建 Meegle Production Bug`

操作方式：

1. 打开一条 Lark 记录详情
2. 确认该记录字段完整，尤其是 `Issue 类型`
3. 点击页面里的 `创建 Meegle ...` 按钮
4. 插件把当前记录快照发给后台
5. 后台调用 `/api/lark-base/create-meegle-workitem`
6. 成功后页面会提示 `已创建 ...`

常见失败原因：

- `Meegle` 尚未授权
- 当前页面不是受支持的 Lark 记录详情页
- 记录字段不足以映射到目标工作项类型

### 5.2 在特定 Lark Base 视图中批量创建

如果你打开的是插件当前配置好的目标 Base / Table / View，`自动化` 页签里会额外出现一个按钮：

- `批量创建 MEEGLE TICKET`

操作方式：

1. 打开指定的 Lark Base 视图
2. 打开插件，进入 `自动化`
3. 点击 `批量创建 MEEGLE TICKET`
4. 先查看预览结果
5. 再确认执行批量创建

批量预览会把记录分成几类：

- 可创建
- 已跳过
- 执行失败

适合在一批需求或问题已经整理好的场景下集中建单。

## 6. Meegle 反推 Lark

在 `Meegle` 工作项详情页，插件 `自动化` 页签里会显示：

- `更新Lark及推送`

这个动作的目标是把 Meegle 当前工作项的处理结果反推回 Lark。

当前实现会尽量完成以下动作：

- 更新对应 `Lark Base` 记录状态
- 往 Lark 消息线程发送更新消息
- 给对应消息加 reaction
- 回写 Meegle 工作项自己的“已更新”状态

操作方式：

1. 打开一条 Meegle 工作项详情
2. 打开插件，进入 `自动化`
3. 点击 `更新Lark及推送`
4. 等待插件完成推送并跳转到 `聊天` 页签

前置条件：

- 该工作项里已经有可用的 `Lark Record Link`
- 该工作项里已经有 `Lark Update Message`
- 该工作项里已经有 `Lark Message Link`

如果这些字段缺失，插件会直接报错并停止，不会做部分成功的伪完成提示。

## 7. GitHub PR 反查 Meegle

当你打开 `GitHub Pull Request` 页面时，插件会把页面识别为 `GitHub` 平台，并在 `自动化` 页签里显示：

- `查询 PR 关联的 Meegle 工作项`

操作方式：

1. 打开一个 GitHub PR 页面
2. 打开插件，进入 `自动化`
3. 点击 `查询 PR 关联的 Meegle 工作项`
4. 等待结果返回

当前实现会从以下内容里提取 Meegle ID：

- PR 标题
- PR 描述
- commit message
- issue comments
- review comments

查询结果会在插件里直接展示：

- 提取到了哪些 Meegle ID
- 找到了哪些工作项
- 哪些 ID 没有查到
- 已找到工作项的状态、类型、计划版本、计划迭代和跳转链接

前置条件：

- 当前页面必须是 `github.com/<owner>/<repo>/pull/<number>`
- 服务端已经配置 `GITHUB_TOKEN`
- 当前用户的 Meegle 授权仍然有效

## 8. Meegle 创建 GitHub 分支

当你打开 `Meegle` 工作项详情页时，插件 `自动化` 页签里还会显示：

- `创建 GitHub 分支`

操作方式：

1. 打开一条 Meegle 工作项详情
2. 打开插件，进入 `自动化`
3. 点击 `创建 GitHub 分支`
4. 插件先请求预览信息
5. 在弹窗里确认仓库、系统、工作项标题和默认分支名
6. 如有需要，手动编辑分支名
7. 点击 `确认创建`

创建成功后，弹窗会返回：

- 新分支名称
- GitHub 跳转链接

补充说明：

- 当前分支名输入限制为 50 个字符
- 具体仓库和默认分支名由服务端根据工作项信息生成
- 如果服务端没有配置 `GITHUB_TOKEN`，该动作会失败

## 9. AI 聊天

插件右侧有独立的 `聊天` 页签。该页签和 `自动化` 共用授权前提，必须在 `Lark` 与 `Meegle` 都授权完成后才能正常使用。

### 9.1 打开方式

有两种常见打开方式：

- 直接点击右侧 `聊天` 页签
- 在 `自动化` 页签点击 `分析当前页面`

当你在 `自动化` 里点击 `分析当前页面` 时：

- 如果聊天还没打开，插件会切到聊天页
- 如果当前已有会话，再点一次会重置当前会话

### 9.2 能力说明

当前聊天页接入的是服务端 `ACP Kimi` 流式接口，支持：

- 多轮追问
- 流式返回
- 历史会话列表
- 打开旧会话
- 删除旧会话
- 手动停止生成

你可以把它理解为一个围绕当前协作场景的插件内 AI 助手，适合做：

- 当前页面信息分析
- 跟进问题补充
- 结果解释
- 后续行动建议

### 9.3 使用建议

- 在有明确上下文的页面里再打开聊天，效果更稳定
- 如果当前对话已经跑偏，直接重新点一次 `分析当前页面` 重开会话
- 重要结果仍建议回到原始 Lark / Meegle / GitHub 页面复核

## 11. 常见问题

### 11.1 自动化页签是灰的

通常是因为 `Lark` 或 `Meegle` 还没有完成授权。先到 `个人` 页签完成两边授权。

### 11.2 Meegle 授权老是失败

优先检查：

- 当前标签页是否就是已登录的 Meegle 页面
- 服务端是否已正确下发 `MEEGLE Plugin ID`
- 服务端 Meegle 配置是否完整

### 11.3 GitHub PR 反查不到结果

优先检查：

- PR 标题、描述、commit、评论里是否真的包含 Meegle ID
- 当前服务端是否配置了 `GITHUB_TOKEN`
- 当前用户的 Meegle token 是否还有效

### 11.4 更新横幅出现了，但点完没自动升级

这是当前实现的正常行为。插件只负责检查和下载，最终仍需要你手动去 `chrome://extensions/` 替换安装包。

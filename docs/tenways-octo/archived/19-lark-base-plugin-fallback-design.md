# Lark Base 多维表格插件替补方案

## 1. 背景

当前 Tenways Octo 的主入口仍然是浏览器插件：

- 插件负责触发、上下文采集、结果展示
- 服务端负责身份、授权、字段映射、草稿生成、创建 Meegle 工作项

在 Lark 多维表格场景下，现有思路更偏向于：

- 在当前网页中探测列表 / 详情态
- 通过 content script 注入按钮
- 读取当前记录上下文并调用服务端

这条路可以继续做，但有一个明显风险：多维表格页面是高动态 UI，存在视图切换、分组、虚拟滚动、详情抽屉重建、DOM 结构调整等问题。尤其“在原生表格每一行插入按钮”这件事，稳定性和维护成本都偏高。

因此需要准备一条替补方案：

- 不依赖 Chrome 对原生 DOM 的深度注入
- 改为使用 Lark / 飞书官方多维表格插件能力
- 在插件面板或侧栏中根据当前选中的 `recordId` 自动识别“当前记录”
- 提供 `发送到 Meegle` 操作

## 2. 方案定位

该方案是 **替补方案**，不是对当前浏览器插件主架构的全量替代。

目标是：

- 在 Lark 多维表格场景下，提供一个更稳定的官方入口
- 降低对多维表格 DOM 结构的耦合
- 保持服务端业务编排不变

不在本方案中处理：

- 替换整个 Chrome 扩展
- 重做当前 Meegle 授权模型
- 多租户、多项目、多目标类型的复杂配置中心
- 原生行内按钮的官方插件注入能力

## 3. 设计结论

### 3.1 推荐入口

推荐使用 **Lark Base 多维表格插件面板 / 侧栏** 作为入口。

交互主线：

1. 用户在多维表格中点击一条记录
2. 官方插件通过 Base JS SDK 识别当前 `tableId / viewId / recordId`
3. 插件面板展示当前记录摘要
4. 用户点击 `发送到 Meegle`
5. 服务端生成 Meegle 草稿
6. 插件面板展示映射结果与确认信息
7. 用户确认后创建到 Meegle
8. 插件面板返回成功状态、Meegle 链接、可选回写信息

### 3.2 不采用的主入口

不把“在原生表格每一行或原生详情头部直接插入自定义按钮”作为正式主路。

原因：

- 当前查到的官方插件文档可以确认插件容器、选择态监听、记录读取、UI 交互能力
- 但没有明确文档依据证明“可以稳定把自定义按钮插进原生每一行 / 原生详情头部”
- 即使技术上可以靠 DOM hack 实现，也仍然回到了最初想规避的高耦合方案

因此该方案以 **官方插件面板识别当前记录** 为边界。

## 4. 官方能力边界

根据 Base JS SDK 文档，已确认可用的关键能力包括：

- 获取当前选择：`bitable.base.getSelection()`
- 监听选择变化：`bitable.base.onSelectionChange(...)`
- 获取当前活动表：`bitable.base.getActiveTable()`
- 按 `recordId` 获取记录：`table.getRecordById(recordId)`
- 获取视图字段元数据：`view.getFieldMetaList()`
- 获取当前可见记录：`view.getVisibleRecordIdList()`
- 获取当前用户 / 租户：`bitable.bridge.getUserId()`、`bitable.bridge.getTenantKey()`
- 弹出提示 / 详情：`bitable.ui.showToast()`、`bitable.ui.showRecordDetailDialog(...)`

这些能力足够支持：

- 识别当前操作记录
- 读取当前记录字段
- 在插件内展示草稿与确认态
- 将数据发送到自有服务端

## 5. 用户交互设计

### 5.1 空态

当用户还没有选中任何记录时，插件面板展示：

- 文案：`请先在多维表格中选择一条记录`
- 辅助说明：`插件会自动识别当前选中的记录，并生成 Meegle 草稿`

### 5.2 已选中记录态

当 SDK 能拿到稳定的 `recordId` 后，插件面板展示一个“当前记录卡片”：

- 标题
- 关键编号 / 单号
- Requester / PM / 优先级 / 状态
- 描述摘要
- 按钮：`发送到 Meegle`

这一步不要求用户在插件里二次选择记录。

### 5.3 草稿态

点击 `发送到 Meegle` 后：

- 插件把 `master_user_id`、`tenant_key`、`tableId`、`viewId`、`recordId` 发给服务端
- 服务端拉取 / 接收记录字段，生成 Meegle 草稿
- 面板显示草稿确认区

确认区建议展示：

- Meegle 目标项目
- 工作项类型
- 标题
- 描述
- 优先级映射
- 负责人 / 产品经理映射
- 缺失字段提醒

按钮：

- `返回`
- `确认创建`

### 5.4 成功态

创建成功后，面板展示：

- `已发送到 Meegle`
- Meegle 工作项编号
- 打开链接按钮
- 可选的“回写当前记录”结果

### 5.5 失败态

失败时不做静默降级，直接给出明确原因：

- 未授权
- 目标项目未配置
- 当前记录缺少必填字段
- Meegle 创建失败

并提供：

- `重试`
- `查看详细错误`

## 6. 数据流

### 6.1 识别当前记录

官方插件前端：

1. 调用 `bitable.base.getSelection()`
2. 拿到当前 `tableId / viewId / recordId`
3. 如果无 `recordId`，显示空态
4. 如果有 `recordId`，读取当前记录并渲染记录卡片
5. 同时监听 `onSelectionChange(...)`，在用户切换记录时刷新卡片内容

### 6.2 发送到服务端

前端请求建议为：

`POST /api/lark/base/transport/draft`

请求体建议：

```json
{
  "masterUserId": "usr_xxx",
  "tenantKey": "tenant_xxx",
  "baseId": "bascn_xxx",
  "tableId": "tbl_xxx",
  "viewId": "vew_xxx",
  "recordId": "rec_xxx",
  "recordSnapshot": {
    "title": "...",
    "description": "...",
    "priority": "...",
    "requester": "...",
    "pm": "...",
    "status": "..."
  }
}
```

服务端返回：

- 标准化字段
- Meegle 草稿
- 缺失字段
- 是否可直接创建

确认创建再走：

`POST /api/lark/base/transport/apply`

## 7. 与现有系统的关系

### 7.1 保持不变的部分

以下部分不需要因为该替补方案而重写：

- `users / user_tokens` 身份与授权模型
- Lark OAuth callback 机制
- Meegle token 交换与刷新
- Meegle work item 创建服务
- 服务端字段映射与草稿生成逻辑

### 7.2 需要新增的部分

新增一套 **Lark Base 插件前端**：

- 单独的前端工程或子入口
- 运行在 Lark Base 插件容器内
- 通过 Base JS SDK 采集当前记录上下文
- 调用同一个 Tenways Octo 服务端

也就是说，最终可能会同时存在两个 Lark 入口：

1. Chrome 扩展入口
2. Lark Base 官方插件入口

两者共享同一套服务端业务能力。

## 8. 身份与授权处理

### 8.1 Lark 身份

Lark Base 插件内可以直接拿到：

- 当前用户 ID
- 当前租户 Key

因此该入口在身份上反而比纯 DOM 探测更干净。

推荐处理：

- 插件启动时先调用服务端 `identity/resolve`
- 用 `tenantKey + larkUserId` 获得或确认 `master_user_id`
- 后续所有草稿 / 创建请求都带 `master_user_id`

### 8.2 Meegle 授权

该替补方案不改变当前 Meegle 授权模型。

如果服务端判断当前用户没有可用的 Meegle token，则插件面板应展示：

- `Meegle 未授权`
- 引导用户回到主插件或指定入口完成授权

后续如果要进一步收敛，也可以考虑让 Lark Base 插件入口复用统一的授权中转页，但这不属于本方案首轮范围。

## 9. 一期实现边界

一期建议只支持：

- 单条记录搬运到 Meegle
- 单一目标项目 / 单一工作项类型
- 固定字段映射模板
- 面板内草稿确认
- 成功后展示结果，不做复杂回写

不建议一期就做：

- 批量搬运
- 多项目动态切换
- 复杂模板管理
- 回写多个字段并维护双向同步

## 10. 风险与取舍

### 10.1 优点

- 官方能力更稳定
- 不依赖复杂 DOM 注入
- 获取当前记录的方式更清晰
- 更适合在多维表格内长期维护

### 10.2 代价

- 需要单独维护一个 Lark Base 插件前端
- 需要补一套插件发布 / 运行环境
- 用户入口会从“Chrome 扩展唯一入口”变成“扩展 + Base 插件双入口”

### 10.3 关键取舍

本方案本质上是：

- 用一套额外入口，换取 Lark 多维表格内更稳定的交互

这不是更小的实现，而是更稳的实现。

## 11. 推荐落地顺序

1. 先保留现有 Chrome 扩展主线
2. 新建 Lark Base 插件 POC
3. 先跑通：
   - 获取当前 `recordId`
   - 读取记录摘要
   - 调用服务端生成草稿
   - 确认创建到 Meegle
4. 验证用户是否接受“面板式入口”
5. 再决定是否将该入口升级为 Lark 场景主入口

## 12. 最终结论

对于“在 Lark 多维表格中选中一条记录，发送到 Meegle”这个场景，推荐准备一条官方插件替补方案：

- 不追求原生行内注入
- 使用 Lark Base 插件面板 / 侧栏
- 依据当前 `recordId` 自动识别当前记录
- 在面板内完成草稿确认与创建

如果当前 Chrome 注入方案在真实多维表格中出现稳定性问题，这条方案应作为优先替补路线推进。
